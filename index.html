O que é este projeto?

Visão geral do sistema em linguagem simples

Este é um sistema de gerenciamento de academia desenvolvido em Java, com persistência de dados em PostgreSQL. Ele roda no terminal (linha de comando) e permite que um funcionário da academia gerencie tudo: alunos, instrutores, planos, aulas coletivas e controle de presença.

Imagine que você é recepcionista de uma academia. Antes deste sistema, você usava papel e caneta pra tudo — anotar quem se matriculou, qual plano escolheu, em quais aulas está inscrito, quando entrou na academia. Este sistema substitui exatamente esse papel e caneta, mas com validações automáticas, banco de dados e código orientado a objetos.

🏋️
Alunos
Cadastro completo com nome, CPF, e-mail, data de nascimento, data de matrícula e plano vinculado. O sistema calcula automaticamente se o plano está ativo ou vencido.
🧑‍🏫
Instrutores
Cada instrutor tem especialidade e horários de trabalho. Podem ser vinculados às aulas coletivas que ministram.
📋
Planos
Básico (1 mês), Intermediário (3 meses) ou Premium (6 meses). O sistema usa a duração do plano para calcular o vencimento automaticamente.
🧘
Aulas & Inscrições
Aulas coletivas com capacidade máxima e horário definido. A inscrição passa por 5 validações antes de ser confirmada.

Encapsulamento CP1

Proteger os dados da classe do mundo externo

Encapsulamento é o princípio de esconder os detalhes internos de uma classe e expor apenas o que é necessário. Em Java, isso é feito com modificadores de acesso (private, public, protected) e com getters e setters.

Imagine seu CPF. Ele é um dado privado — você não sai exibindo para qualquer um. Mas se alguém precisar saber, você fornece de forma controlada. Em Java: o atributo é private, e você fornece acesso via getCpf(). E se alguém quiser alterar? Só pela função setCpf(), onde você pode inclusive validar antes de aceitar.

Como está no projeto

Veja a classe Plano.java — um exemplo perfeito de encapsulamento:

model/Plano.java Java
public class Plano { // Atributos PRIVADOS — ninguém acessa diretamente de fora private int id; private String nome; private double valorMensal; private int duracaoMeses; // ex: 1, 3 ou 6 // Para LER o nome: usa o getter public String getNome() { return nome; } // Para ALTERAR o nome: usa o setter (aqui poderia ter validação) public void setNome(String nome) { this.nome = nome; } // Getter da duração (usada para calcular vencimento do plano) public int getDuracaoMeses() { return duracaoMeses; } }
✅
Por que private? Se os atributos fossem public, qualquer parte do código poderia fazer plano.valorMensal = -500 e quebrar a consistência dos dados. Com private + setter, você pode validar: "valorMensal não pode ser negativo".

ValidadorUtil — encapsulamento de regras

util/ValidadorUtil.java Java
public class ValidadorUtil { // Valida CPF: deve ter exatamente 11 dígitos public static boolean validarCpf(String cpf) { if (cpf == null) return false; String limpo = cpf.replaceAll("[^0-9]", ""); // remove pontos e traços return limpo.length() == 11; } // Valida e-mail: precisa ter @ e . public static boolean validarEmail(String email) { return email != null && email.contains("@") && email.contains("."); } }

Herança CP2

Reutilizar código através de hierarquias de classes

Herança permite que uma classe filho herde os atributos e métodos de uma classe pai. Em Java, usa-se a palavra-chave extends.

Aluno, Instrutor e Funcionário são todos Pessoas. Todos têm nome, CPF e telefone. Em vez de repetir esses três atributos em cada classe, criamos uma classe-pai Pessoa com eles, e cada classe-filha herda automaticamente. É como dizer: "tudo que uma Pessoa tem, um Aluno também tem — mais as coisas específicas dele."

Pessoa (superclasse abstrata) ├── Aluno ← extends Pessoa, implements Auditavel │ + email, dataNascimento, dataMatricula, plano │ + isPlanoAtivo(), getDataVencimentoPlano() ├── Instrutor ← extends Pessoa, implements Auditavel │ + especialidade, horariosTrabalho, salario └── Funcionario ← extends Pessoa + cargo, turno, salario

A superclasse Pessoa

model/Pessoa.java Java
public abstract class Pessoa { // Atributos COMUNS a todos (Aluno, Instrutor, Funcionario) protected int id; protected String nome; protected String cpf; protected String telefone; // Construtor: chamado pelo super() nas subclasses public Pessoa(int id, String nome, String cpf, String telefone) { this.id = id; this.nome = nome; this.cpf = cpf; this.telefone = telefone; } // Método ABSTRATO: obriga cada subclasse a ter sua própria implementação public abstract void exibirDetalhes(); // Método ABSTRATO: cada pessoa tem um custo diferente para a academia public abstract double calcularCustoMensal(); // Método CONCRETO: compartilhado por todos, sem precisar reimplementar public void exibirInformacoesBasicas() { System.out.println("ID: " + id + " | Nome: " + nome + " | CPF: " + cpf); } }

Como Aluno usa a herança

model/Aluno.java (trecho) Java
// "extends Pessoa" = Aluno herda id, nome, cpf, telefone // "implements Auditavel" = Aluno se compromete a ter registrarLog e obterHistorico public class Aluno extends Pessoa implements Auditavel { // Atributos ESPECÍFICOS de Aluno (além dos herdados de Pessoa) private String email; private LocalDate dataMatricula; private Plano plano; public Aluno(int id, String nome, String cpf, String telefone, String email, LocalDate dataNascimento, LocalDate dataMatricula, Plano plano) { // Chama o construtor de Pessoa para inicializar id, nome, cpf, telefone super(id, nome, cpf, telefone); this.email = email; this.dataMatricula = dataMatricula; this.plano = plano; } // Implementação do método abstrato de Pessoa @Override public void exibirDetalhes() { System.out.println("--- Detalhes do Aluno ---"); exibirInformacoesBasicas(); // método herdado de Pessoa System.out.println("Email: " + email); System.out.println("Status do Plano: " + (isPlanoAtivo() ? "✅ Ativo" : "❌ Vencido")); } // Implementação do método abstrato de Pessoa @Override public double calcularCustoMensal() { return 0; // Aluno é receita, não custo para a academia } }
ℹ️
O super(id, nome, cpf, telefone) é obrigatório quando a superclasse não tem construtor sem argumentos. Ele diz: "antes de configurar as coisas específicas do Aluno, inicializa os atributos de Pessoa."

Polimorfismo CP2

O mesmo método, comportamentos diferentes

Polimorfismo vem do grego: "muitas formas". Na POO, significa que o mesmo método pode ter comportamentos diferentes dependendo de qual objeto o está executando.

Imagine que você manda uma mensagem de texto "Qual é o seu custo mensal?" para três funcionários diferentes da academia. O Instrutor responde "R$ 2.500,00". O Funcionário responde "R$ 1.800,00". O Aluno responde "R$ 0,00 — eu sou receita, não custo!". Todos respondem à mesma pergunta (calcularCustoMensal()), mas cada um responde de forma diferente. Isso é polimorfismo.

Sobrescrita de método com @Override

Polimorfismo em ação Java
// Pessoa é abstrata, mas podemos usar como tipo de variável ArrayList<Pessoa> pessoas = new ArrayList<>(); // Adicionamos objetos de tipos DIFERENTES na mesma lista pessoas.add(new Aluno(...)); // calcularCustoMensal() → 0.0 pessoas.add(new Instrutor(...)); // calcularCustoMensal() → 2500.0 pessoas.add(new Funcionario(...)); // calcularCustoMensal() → salario // Para cada pessoa, Java chama o método CORRETO automaticamente for (Pessoa p : pessoas) { p.exibirDetalhes(); // cada um exibe suas infos específicas p.calcularCustoMensal(); // cada um retorna seu próprio valor }
✅
O @Override não é obrigatório, mas é uma boa prática. Ele diz ao compilador: "estou intencionalmente sobrescrevendo um método da superclasse". Se você errar o nome do método, o compilador vai avisar em vez de criar um método novo silenciosamente.

Sobrecarga (Overloading) — diferente de sobrescrita

A sobrecarga é quando você tem dois métodos com o mesmo nome mas parâmetros diferentes na mesma classe. Isso também foi implementado em Aluno.

Sobrecarga vs Sobrescrita Java
// SOBRESCRITA (Override) — mesmo método, classe diferente // Pessoa define: public void exibirDetalhes() // Aluno sobrescreve com @Override // SOBRECARGA (Overloading) — mesmo método, parâmetros diferentes, mesma classe // calcularMensalidade() → sem desconto // calcularMensalidade(double desconto) → com desconto aplicado public double calcularMensalidade() { return plano.getValorMensal(); } public double calcularMensalidade(double desconto) { return plano.getValorMensal() * (1 - desconto); // ex: 10% de desconto }

Classe Abstrata CP3

Classes que existem apenas para ser herdadas

Uma classe abstrata é uma classe que não pode ser instanciada — você não pode fazer new Pessoa(). Ela existe apenas para ser a base de outras classes. Pode ter métodos concretos (implementados) e métodos abstratos (que as subclasses são obrigadas a implementar).

Pense em "Veículo". Não existe um veículo genérico — existe carro, moto, ônibus. "Veículo" é um conceito abstrato. Da mesma forma, "Pessoa" no nosso sistema é abstrata: na academia real, toda pessoa é especificamente aluno, instrutor ou funcionário. Não existe uma "Pessoa genérica" cadastrada.

Tipo de Método Na Pessoa Nas Subclasses
abstract exibirDetalhes() Declarado, não implementado Obrigatoriamente implementado com @Override
abstract calcularCustoMensal() Declarado, não implementado Cada subclasse retorna seu próprio valor
exibirInformacoesBasicas() Implementado (concreto) Herdado, pode ser chamado com super ou diretamente
⚠️
Diferença crucial: Classe abstrata usa extends e uma classe só pode herdar de uma classe abstrata. Interface usa implements e uma classe pode implementar várias interfaces. É uma das perguntas favoritas de professor.

Interfaces CP4 ★

Contratos que garantem que uma classe terá determinados comportamentos

Uma interface é um contrato. Ela define quais métodos uma classe deve ter, mas não diz como implementá-los. É como assinar um contrato de prestação de serviço: você se compromete a entregar o que foi combinado, mas a forma de fazer é sua.

Imagine uma tomada de padrão NBR (o padrão brasileiro). Qualquer aparelho que tenha o plugue correto funciona nela — não importa se é um liquidificador ou um notebook. A interface é o padrão da tomada: define o "formato", e qualquer classe que "implementa" essa interface garante que vai se encaixar.

No projeto: a interface Auditavel diz "quem me implementa garante ter registrarLog() e obterHistorico()". Tanto Aluno quanto Instrutor "se encaixam" nessa tomada.

Interface Auditavel

util/Auditavel.java Java
public interface Auditavel { // Registra uma ação realizada sobre a entidade // Exemplo: "INSCRIÇÃO na aula Yoga Matinal [ID 1]" void registrarLog(String acao); // Retorna o histórico completo de ações String obterHistorico(); }

Como Aluno implementa Auditavel

model/Aluno.java (trecho — Auditavel) Java
public class Aluno extends Pessoa implements Auditavel { // Lista interna de logs — privada, ninguém acessa direto private final List<String> historico = new ArrayList<>(); @Override public void registrarLog(String acao) { // Monta a entrada com timestamp + identificação + ação String entrada = LocalDateTime.now() + " | ALUNO[" + id + "] | " + acao; historico.add(entrada); System.out.println("📝 Log: " + entrada); } @Override public String obterHistorico() { if (historico.isEmpty()) return "Nenhum registro."; StringBuilder sb = new StringBuilder("=== HISTÓRICO DO ALUNO " + nome + " ===\n"); historico.forEach(log -> sb.append(" ").append(log).append("\n")); return sb.toString(); } }

Interface Relatorio

util/Relatorio.java Java
public interface Relatorio { // Exibe o relatório formatado no console void gerarRelatorio(); // Exporta o relatório para um arquivo de texto void exportarParaArquivo(String nomeArquivo); }

A classe RelatorioService implementa essa interface. Ela gera um relatório completo com: contagem de alunos ativos/vencidos, lista detalhada de cada aluno com visitas e aulas inscritas, ocupação de vagas por aula e lista de instrutores.

Pode ter atributos? Sim Não (só constantes)

Padrão DAO CP3

Data Access Object — separar o código Java do banco de dados

DAO é um padrão de projeto que cria uma camada dedicada para todas as operações de banco de dados. O resto do sistema faz pedidos ao DAO sem saber como o banco funciona por baixo.

Imagine um bibliotecário. Você não entra no estoque para pegar um livro você mesmo — você pede ao bibliotecário: "me traz o livro de ID 5". Ele sabe onde estão os livros, como buscá-los, e te entrega o resultado. O DAO é esse bibliotecário entre seu código Java e o banco PostgreSQL.

Estrutura padrão de um DAO

Todo DAO do projeto segue o mesmo padrão de 4 operações (CRUD):

dao/PlanoDAO.java (estrutura simplificada) Java
public class PlanoDAO { // CREATE — Insere um novo plano no banco public boolean inserir(Plano plano) { String sql = "INSERT INTO plano (nome, valor_mensal, duracao_meses) VALUES (?, ?, ?)"; try (Connection conn = ConexaoBD.getConexao(); PreparedStatement stmt = conn.prepareStatement(sql)) { stmt.setString(1, plano.getNome()); stmt.setDouble(2, plano.getValorMensal()); stmt.setInt(3, plano.getDuracaoMeses()); return stmt.executeUpdate() > 0; } catch (SQLException e) { System.err.println("Erro: " + e.getMessage()); } return false; } // READ — Lista todos os planos public List<Plano> listarTodos() { /* SELECT * FROM plano */ } // READ — Busca plano por ID public Plano buscarPorId(int id) { /* SELECT * FROM plano WHERE id = ? */ } // UPDATE — Atualiza os dados de um plano public boolean atualizar(Plano plano) { /* UPDATE plano SET ... WHERE id = ? */ } // DELETE — Remove um plano public boolean excluir(int id) { /* DELETE FROM plano WHERE id = ? */ } }
ℹ️
O try-with-resources (o try (Connection conn = ...)) garante que a conexão com o banco seja fechada automaticamente ao final, mesmo que ocorra um erro. Isso evita vazamento de conexões — um problema grave em sistemas reais.

PreparedStatement — segurança contra SQL Injection

Por que usar ? em vez de concatenar strings Java
// ❌ PERIGOSO — SQL Injection! Se nome = "'; DROP TABLE aluno; --" String sql = "SELECT * FROM aluno WHERE nome = '" + nome + "'"; // ✅ SEGURO — PreparedStatement escapa os valores automaticamente String sql = "SELECT * FROM aluno WHERE nome = ?"; stmt.setString(1, nome); // o ? vira o valor, sem risco de injeção

JDBC e ConexaoBD CP3

Como o Java se conecta ao PostgreSQL

JDBC (Java Database Connectivity) é a API padrão do Java para se comunicar com bancos de dados relacionais. O projeto usa o driver postgresql-xx.jar no classpath.

dao/ConexaoBD.java Java
public class ConexaoBD { private static String url, user, password; // Bloco estático: executa UMA VEZ quando a classe é carregada static { Properties props = new Properties(); try (FileInputStream fis = new FileInputStream("config.properties")) { props.load(fis); url = props.getProperty("db.url"); // jdbc:postgresql://localhost:5432/academia_db user = props.getProperty("db.user"); // postgres password = props.getProperty("db.password"); // sua_senha } catch (IOException e) { // Fallback: valores padrão se o arquivo não existir url = "jdbc:postgresql://localhost:5432/academia_db"; user = "postgres"; password = "root"; } } public static Connection getConexao() throws SQLException { try { // Carrega o driver JDBC do PostgreSQL Class.forName("org.postgresql.Driver"); return DriverManager.getConnection(url, user, password); } catch (ClassNotFoundException e) { throw new SQLException("Driver não encontrado. Coloque o .jar no classpath."); } } }
✅
As credenciais ficam no arquivo config.properties, que está no .gitignore. Assim a senha nunca vai para o repositório. Existe um config.properties.example com os campos sem a senha real — bom exemplo de segurança em projetos reais.

Regra de Negócio Complexa CP4 ★

InscricaoService — 5 validações em cascata antes de confirmar uma inscrição

Esta é a parte mais importante do CP4. O InscricaoService implementa a lógica completa de inscrição de um aluno em uma aula, realizando 5 verificações em sequência. Se qualquer uma falhar, a inscrição é recusada com uma mensagem explicativa.

1
Aluno existe no banco?
Chama alunoDAO.buscarPorId(idAluno). Se retornar null, o aluno não existe.
❌ "Aluno não encontrado (ID X)"
2
Plano do aluno está ativo?
Chama aluno.isPlanoAtivo() — que calcula dataMatricula + duracaoMeses e compara com hoje.
❌ "Plano vencido em [data]. Renove para se inscrever."
3
Aula existe no banco?
Chama aulaDAO.buscarPorId(idAula). Valida antes de qualquer outra operação relacionada à aula.
❌ "Aula não encontrada (ID X)"
4
Há vagas disponíveis?
Conta inscritos com inscricaoDAO.contarInscritosNaAula() e compara com aula.getCapacidadeMaxima().
❌ "Capacidade máxima atingida (X/X alunos)"
5
Conflito de horário?
Busca se o aluno já tem outra aula no mesmo horário: inscricaoDAO.buscarConflitoHorario(idAluno, aula.getHorario()).
❌ "Conflito com aula [Nome] no mesmo horário"
✓
Todas as validações passaram!
Persiste a inscrição no banco, registra log de auditoria via aluno.registrarLog(...) e retorna mensagem de sucesso.

O código completo do método

service/InscricaoService.java — inscreverAluno() Java
public String inscreverAluno(int idAluno, int idAula) { // VALIDAÇÃO 1: O aluno existe no banco? Aluno aluno = alunoDAO.buscarPorId(idAluno); if (aluno == null) { return "❌ Erro: Aluno não encontrado (ID " + idAluno + ")."; } // VALIDAÇÃO 2: O plano está dentro do prazo? // isPlanoAtivo() faz: LocalDate.now().isAfter(dataMatricula.plusMonths(duracao)) if (!aluno.isPlanoAtivo()) { return "❌ Plano de " + aluno.getNome() + " vencido em " + aluno.getDataVencimentoPlano() + ". Renove para se inscrever."; } // VALIDAÇÃO 3: A aula existe? Aula aula = aulaDAO.buscarPorId(idAula); if (aula == null) { return "❌ Erro: Aula não encontrada (ID " + idAula + ")."; } // VALIDAÇÃO 4: Ainda tem vagas? int inscritos = inscricaoDAO.contarInscritosNaAula(idAula); if (inscritos >= aula.getCapacidadeMaxima()) { return "❌ Capacidade máxima atingida (" + inscritos + "/" + aula.getCapacidadeMaxima() + ")."; } // VALIDAÇÃO 5: Tem conflito de horário? Aula conflito = inscricaoDAO.buscarConflitoHorario(idAluno, aula.getHorario()); if (conflito != null) { return "❌ Conflito! Aluno já inscrito em \"" + conflito.getNome() + "\" no mesmo horário."; } // TUDO OK — persiste no banco boolean sucesso = inscricaoDAO.salvarInscricao(idAluno, idAula); if (sucesso) { // Registra auditoria usando a interface Auditavel aluno.registrarLog("INSCRIÇÃO na aula \"" + aula.getNome() + "\" [ID " + idAula + "]"); return "✅ Sucesso: " + aluno.getNome() + " inscrito em \"" + aula.getNome() + "\"."; } return "❌ Erro inesperado. Tente novamente."; }

Como isPlanoAtivo() funciona

Perguntas Prováveis do Professor

Prepare respostas claras para cada uma

Por que Pessoa é abstrata e não uma classe normal?
Porque no contexto do nosso sistema, não faz sentido existir uma "Pessoa genérica" cadastrada. Toda pessoa é especificamente aluno, instrutor ou funcionário. A abstração também obriga cada subclasse a implementar exibirDetalhes() e calcularCustoMensal(), garantindo que nunca esquecemos de implementar esses métodos em uma nova subclasse.
Qual a diferença entre classe abstrata e interface no projeto?
Pessoa é classe abstrata: compartilha código real (atributos id, nome, cpf e o método concreto exibirInformacoesBasicas()). Uma subclasse só pode herdar de uma classe abstrata. Auditavel é interface: só define o contrato (quais métodos devem existir), sem implementação. Uma classe pode implementar várias interfaces simultaneamente — Aluno implementa Auditavel enquanto também herda de Pessoa.
Por que a regra de negócio está no InscricaoService e não no DAO?
Separação de responsabilidades. O DAO só sabe persistir e buscar dados — ele não deve saber se um plano está vencido ou se há conflito de horário. O Service centraliza a lógica de negócio. Se amanhã a regra mudar (ex: alunos Premium podem se inscrever mesmo com plano vencido), mudamos só o Service, sem tocar o DAO ou o model.
O que é polimorfismo e onde ele aparece no projeto?
Polimorfismo é quando objetos de tipos diferentes respondem ao mesmo método de formas diferentes. No projeto: calcularCustoMensal() existe em Pessoa, mas Aluno retorna 0, Instrutor retorna o salário (R$2.500) e Funcionario retorna o salário cadastrado. Se fizermos um ArrayList<Pessoa> com os três, ao chamar calcularCustoMensal() em cada um, Java chama automaticamente a versão correta para cada tipo.
Por que usar PreparedStatement em vez de concatenar strings SQL?
Segurança. Concatenar strings SQL permite SQL Injection — um atacante pode digitar um nome como ' OR 1=1 -- e manipular a query. O PreparedStatement usa ? como placeholder e escapa automaticamente os valores, tornando isso impossível. Além disso, PreparedStatement é pré-compilado pelo banco, sendo mais eficiente.
Por que as interfaces Auditavel e Relatorio foram escolhidas?
Auditavel foi escolhida porque rastreabilidade é um comportamento que classes não relacionadas podem precisar — tanto Aluno quanto Instrutor precisam logar ações, mas eles não têm outra relação entre si. Relatorio define o contrato de geração de relatórios: qualquer classe que precise gerar relatórios implementa essa interface, garantindo que sempre haverá gerarRelatorio() e exportarParaArquivo().
O que é encapsulamento e como está aplicado?
Encapsulamento é proteger os dados internos de uma classe, expondo acesso só através de métodos controlados. No projeto, todos os atributos das classes model são private. O acesso é feito via getters/setters. Isso permite que no futuro, um setter possa incluir validação sem quebrar o resto do código. Exemplo: ValidadorUtil.validarCpf() pode ser chamado dentro de setCpf() para garantir que nenhum CPF inválido seja aceito.
Como funciona o try-with-resources nos DAOs?
O try (Connection conn = ConexaoBD.getConexao(); ...) garante que a conexão com o banco seja fechada automaticamente ao sair do bloco — mesmo se ocorrer uma exceção. Sem isso, conexões ficariam abertas e vazariam recursos, eventualmente esgotando o pool de conexões do banco. É um padrão essencial em código de produção.
O que é o padrão DAO e por que usá-lo?
DAO (Data Access Object) centraliza toda a comunicação com o banco em classes dedicadas. As vantagens são: o código de negócio não mistura SQL com lógica Java; se trocarmos de banco (PostgreSQL → MySQL), mudamos só os DAOs; fica muito mais fácil testar o código em isolamento. No projeto, cada entidade tem seu próprio DAO com os mesmos 4 métodos básicos: inserir, listar, buscarPorId, atualizar, excluir.
